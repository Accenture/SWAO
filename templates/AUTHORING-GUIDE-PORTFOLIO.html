PowerBI Desktop authoring guide -- swao-portfolio.pbit ​

This guide is for consultants who want to customise or rebuild the portfolio template. If you just want to open the template and load your data, see How it works -- PowerBI dashboards instead. Your template is at wsp/templates/powerbi/swao-portfolio.pbit.

The portfolio template is a separate .pbit from the single-app one, not an extension of it. Portfolio assessments (multiple apps in one workspace) use swao-portfolio.pbit; single-app assessments use swao-report.pbit.

v1.3 update -- master/detail pattern, per-app evidence URLs, RAG palette ​

Sprint 030 #0250 + #0251 + #0269 retrofitted the swao-report v1.5 master/detail pattern across the portfolio template. The pattern + the portfolio-specific differences from swao-report are documented in AUTHORING-GUIDE.md Â§6 (master/detail recipe is identical) with two adaptations called out below.

Adaptation A -- portfolio relationships must be M:M ​

In a single-app bundle, fact_controls[control_id] is unique per row. In a portfolio bundle, the same control_id (e.g. GDPR_Art_5) repeats for every assessed app. PBI rejects *:1 cardinality on the bridges and shows "control_id contains duplicate value GDPR_Art_5". Set all four bridge relationships to Viele-zu-Viele (*:*):

The yellow "begrenzte Beziehung" warnings on M:M edges are expected; they signal limited-relationship semantics, which is what we want.

Adaptation C -- defensive measures that handle ghostfolio-style apps ​

Apps assessed without the LLM comp pass produce BLANK for several audit measures. CF rules-based formatting does not match BLANK (BLANK is not a number, so >= -1 is false). To make BLANK render as 0 and pick up the green CF, wrap measures with + 0 or COALESCE:

dax
Negative Signals = CALCULATE(COUNTROWS('fact_signals'), 'fact_signals'[outcome] = "negative") + 0

FP Consideration Rate = DIVIDE([FPs Considered], [Total Signals]) + 0

This makes the "under-assessed" apps visible in the dashboard with honest-zero values rather than blank cells.

Known portfolio-specific limitations ​

  • signal_id as a fourth row level on the matrix triggers InvalidUnconstrainedJoin -- same as on swao-report. Master/detail is the canonical workaround; pin the matrix at app > regime > control and let the Supporting Evidence Tabelle carry the signal context via cross-filter.
  • fact_controls has no GAP rows when only PARTIAL/UNKNOWN verdicts were produced. Use fact_app_heatmap[gap_score] (the severity-weighted risk score) for the Heatmap matrix and KPI cards instead of literal GAP counts.

0. Prerequisites ​

Same as the single-app guide, with one addition: the star/ bundle must be a portfolio bundle at <workspace>/wsp/exports/<ts>/star/ (symmetric dual-wsp, #0230), not a per-app bundle. The portfolio bundle has the same 17 single-app tables PLUS:

  • fact_app_heatmap.csv (one row per app x regime; outcome counts + gap_score)
  • fact_app_summary.csv (one row per app; 7R, scores, totals)

= 19 CSV files total in the portfolio star/ directory.

Generate one with:

swao export --portfolio    # Premium-gated

Or use the test fixture in export-portfolio.test.ts and copy the emitted wsp/exports/<ts>/star/ somewhere durable.

5. DAX measures ​

Carry over the single-app measures, plus:

Total Apps = COUNTROWS('dim_app')

Apps With Critical Risk =
  CALCULATE([Total Apps], 'fact_app_summary'[weighted_risk_score] >= 10)

Average Coverage =
  AVERAGE('fact_app_summary'[coverage_score])

Top Risk App =
  CALCULATE(
    SELECTEDVALUE('dim_app'[app_id]),
    TOPN(1, 'fact_app_summary',
      'fact_app_summary'[weighted_risk_score], DESC)
  )

Negative Signals (Portfolio) =
  SUM('fact_app_summary'[total_negative_signals])

Weighted Risk (Portfolio) =
  SUM('fact_app_summary'[weighted_risk_score])

Apps By 7R =
  COUNTROWS('dim_app')   -- group by [seven_r_label] in visuals

Portfolio Coverage Distribution =
  -- bucket apps by coverage_score; used in histogram on Page 1
  SWITCH(
    TRUE(),
    'fact_app_summary'[coverage_score] >= 0.9, "0.9-1.0",
    'fact_app_summary'[coverage_score] >= 0.8, "0.8-0.9",
    'fact_app_summary'[coverage_score] >= 0.7, "0.7-0.8",

Page 2 update -- measure boolean comparison (v1.2, #0250 .pbit half) ​

The code half of #0250 (Sprint 028 v0.0.32+) emits literal JS booleans for fact_signals[synthesis] and fact_signals[false_positive_considered] instead of strings. The portfolio template's measures must match.

Open the _Measures table and audit every measure that compares against synthesis or false_positive_considered. The change pattern:

Before (v1.1)After (v1.2)
... synthesis = "true"... synthesis = TRUE()
... false_positive_considered = "true"... false_positive_considered = TRUE()

Specifically, audit [FP Consideration Rate], [Synthesis Coverage], and any custom measure derived from these columns. Re-bind to boolean literals; the cells now render as TRUE / FALSE (capitalised) in PowerBI.

Why this matters: v1.1 measures silently returned 0% because "true" = TRUE() is false in DAX. The auditor card "FP Consideration Rate" was showing 0% on bundles that actually had FP-considered signals.

Conditional formatting reload workaround (v1.2, #0250 portfolio half) ​

Sprint 028 audit (reports/0028-Audit-E2E-Solution.md) noticed that some conditional-formatting rules survive .pbit save but unbind on fresh-Desktop reload. Reproduce + workaround:

  1. Open the saved .pbit in a fresh PowerBI Desktop.
  2. Paste the export path; let the data load.
  3. Click each visual that should have CF (Risks card, Auditor coverage cards, "Apps With Critical Risk" gauge); confirm colours render as authored.
  4. If colour rules unbind, the symptom is white cells where the author intended green / amber / red.
  5. Workaround: re-author the CF in the live session, save again as .pbit, repeat the smoke. The second save is the persistent one. Document any rule that needed re-binding in REFRESH-RUNBOOK.md.

Root cause is not fully understood (PowerBI bug suspected -- format strings stored separately from rule references). Workaround is sufficient for v1.2 ship.

PowerBI refresh runbook -- new assessment workflow ​

How to re-run a SWAO assessment and refresh the PowerBI dashboard so it reflects the latest data.

This runbook is for consumers of the .pbit template (consultants, auditors, ops engineers running assessments against client workspaces). For authoring or modifying the template itself, see AUTHORING-GUIDE.md.

First-time users: why the template asks for a folder path ​

When you open swao-report.pbit in PowerBI Desktop, the first thing that happens is a prompt asking for the SWAOExportPath parameter value. This is by design and not a configuration error.

.pbit files are PowerBI templates: they ship with the schema, relationships, DAX measures, and visual layouts -- but no data. Every consumer points the template at their own bundle by pasting the absolute path to their star/ directory.

File typeContainsPrompts for path?
.pbit (template)structure onlyYES -- every time it is opened
.pbix (workbook)structure + cached dataNO -- opens with data already bound

After you open the .pbit once, paste the path, and click Laden, save the result as a .pbix via Datei > Speichern unter. That .pbix carries the data with it; opening it later is one click. This also means each .pbix is a point-in-time snapshot of one specific assessment run, which is useful for audit trails.

Path format: absolute Windows path, no trailing slash, single backslashes (the Beschreibung field in the template shows double-backslashes -- those are escaped doc syntax, not literal). A correct value looks like:

C:\Projects\accenture\swao\examples\portfolio-workspace\portfolio\apps\sovereign-health\wsp\exports\2026-05-10T05-34-42\star

If the path is wrong, PowerBI fails on the first import with "Datei konnte nicht gefunden werden" / "File could not be found" -- redo the parameter via Datei > Optionen und Einstellungen > Datenquelleneinstellungen.

Audience and prerequisites ​

You should follow this runbook when:

  • A new commit landed in the source repo and you want a fresh assessment.
  • A client workshop produced new context inputs (imports/*.csv) that should reshape the analysis.
  • A regulatory regime was updated in the catalogues and you want to re-score the workload against the new controls.
  • You want to compare two runs side-by-side (will need v1.2 multi-run features; for now you save snapshots manually).

Prerequisites:

  • swao CLI installed and on PATH (verify: swao --version).
  • A SWAO workspace with .swao.yml configured (typically under examples/portfolio-workspace/portfolio/apps/<appId>/).
  • PowerBI Desktop installed (Windows; tested with 2.130 / 2024-04 build and later).
  • For real-LLM runs: ANTHROPIC_API_KEY (or equivalent) in env or credential store. For stub runs: nothing extra needed.

1. Re-run the assessment ​

powershell
# Replace <portfolio> with the absolute path to your portfolio workspace
cd <portfolio>

# Run with stub LLM (deterministic, instant, free)
swao assess --app <appId> --llm-stub

# Or with the real LLM configured in .swao.yml
swao assess --app <appId>

# Optional: print per-pass timing table
swao assess --app <appId> --llm-stub --stats

The command writes a new run directory under <portfolio>/apps/<appId>/wsp/runs/<timestamp>/ with run-manifest.json and per-pass YAML files. The wsp/latest.txt pointer is updated to the new run.

Verify success:

powershell
# Check the manifest schema version (should be 1.3 or later for
# v1.0 of the PowerBI template)
$run = Get-Content "<portfolio>\apps\<appId>\wsp\runs\<timestamp>\run-manifest.json" -Raw | ConvertFrom-Json
"schema_version: $($run.schema_version)"
"signals: $($run.total_signals_emitted)"
"weights: $($run.landing_zone_weights | ConvertTo-Json -Compress)"

If schema_version is older than 1.3, your swao CLI is built from older source. Rebuild:

powershell
cd <repo-root>\swao\packages\swao
npm run build

Then re-run the assess.

2. Emit the BI export bundle ​

powershell
swao export --app <appId>

This writes a star-schema CSV bundle under <portfolio>/apps/<appId>/wsp/exports/<timestamp>/. Contents:

manifest.yaml             -- bundle schema version + per-file row counts and SHA-256
README.md                 -- one-paragraph orientation
star/                     -- 17 CSVs (the data PowerBI consumes)
ndjson/                   -- 17 NDJSONs (mirror of star/, for Spark/Trino)
templates/                -- ONLY emitted on Premium tier (.pbit + .twb copies)

Note the absolute path to the star/ directory -- you will need it in step 3.

3. Load the new data into PowerBI ​

You have two options depending on whether you are starting from the template (.pbit) or refreshing an existing report file (.pbix).

Option A -- starting from the template (clean slate) ​

Use this when you want a brand-new report file or are testing the template against a new bundle.

  1. Open swao\docs\templates\powerbi\swao-report.pbit in PowerBI Desktop (Datei > Öffnen).

  2. PowerBI prompts for parameter values. Paste the new star/ path into SWAOExportPath -- including no trailing slash. Example:

    C:\Projects\accenture\swao\examples\portfolio-workspace\portfolio\apps\sovereign-health\wsp\exports\2026-05-10T05-34-42\star

  3. Click Laden. PowerBI imports all 17 CSVs and renders the six pages.

  4. Save as .pbix (Datei > Speichern unter) under a meaningful name -- e.g. sovereign-health-2026-05-10.pbix. You now have a point-in-time snapshot.

Option B -- refreshing an existing .pbix against a new export ​

Use this when you already have a .pbix you have customised and want to point it at a fresh bundle without losing your edits.

  1. Open the existing .pbix in PowerBI Desktop.
  2. Datei > Optionen und Einstellungen > Datenquelleneinstellungen.
  3. Find the entry for the SWAOExportPath parameter (or the absolute folder path if it was hard-coded).
  4. Click Werte ändern and paste the new path.
  5. Click Schließen.
  6. Start > Aktualisieren (Refresh). PowerBI re-imports the CSVs from the new path. All visuals update.
  7. Save the .pbix.

Option C -- automated refresh in PowerBI Service (cloud) ​

Out of scope for v1.0. Once the team has a Premium tenant and a gateway machine with read access to the export path (or to a blob store the export pipeline writes to), schedule refreshes via the service. Document follows in v1.2.

4. Verify the new data is reflected ​

After loading, run this visual check against the Dashboard page:

  1. Last assessed at card -- should show the timestamp of the new run (the Latest Assessment measure reads MAX(fact_runs[finished_at]) so it auto-updates).
  2. 7R verdict / Coverage / signal counts -- compare to the swao assess output you saw in step 1.
  3. Donut "signals by outcome" -- the slice proportions should reflect the new run. If they look identical to the previous run, you may have loaded the wrong path.
  4. Top-5 negative signals table -- the IDs (e.g. INV-05, DATA-01) should match what the assessment summarised in stdout.

If any visual fails to render or shows old data, the most common cause is a stale parameter binding -- redo step 3 and check the parameter value carefully.

5. Schema-bump considerations ​

The bundle has a bundle_schema_version in manifest.yaml. Currently 1.0.0. When this bumps:

BumpAction
1.0.x (patch)No template change needed. Refresh and verify.
1.x.0 (minor)Template still loads, but a new column should be displayed. Author a v1.1 of .pbit (see AUTHORING-GUIDE.md).
2.0.0 (major)Old template will fail to load. Rebuild against the new schema; old .pbix reports must be re-built.

Check bundle_schema_version after each export:

powershell
Select-String -Path "<exports>\<ts>\manifest.yaml" -Pattern "bundle_schema_version"

Bumps are tracked in this repo's swao\docs\releases\ notes.

6. Troubleshooting ​

"File could not be opened" / "Path not found" ​

The parameter path you pasted does not exist on disk, has a typo, or includes Power Query escape characters. Verify with:

powershell
Test-Path "<path-you-pasted>"

If False, fix the path. Common mistakes: leading/trailing whitespace copied from chat, missing \star suffix, double-backslashes from a docs example.

One of the 17 CSV imports fails ​

Most likely cause: the table's first row was not promoted to headers. See AUTHORING-GUIDE.md § 3.1 "Header-row promotion check".

Cards show wrong scale (e.g. 0.16 displayed as 16) ​

A column type was auto-cast to Ganze Zahl (Whole Number) instead of Dezimalzahl. See AUTHORING-GUIDE.md § 3.2 for the gear-icon-on-Geänderter Typ fix; do not click the column-header type dropdown after the integer truncation has happened (it cannot recover the precision).

Total LLM Cost shows "(Leer)" ​

Stub-LLM runs do not populate llm_total_cost_usd. The v1.1 measures wrap this in COALESCE(..., 0) so the card shows 0. If you are on an older .pbit and want the same behaviour, edit the measure.

Visual "Beziehungen verwalten" rejects an activation with "ambiguous paths" ​

See AUTHORING-GUIDE.md § 4.1 "Cycle-breaking" -- deactivate fact_pass_runs (app_id) -> dim_app to break the cycle.

7. Snapshot strategy ​

For audit purposes you may want to keep a .pbix snapshot per sprint-end or per significant client milestone. Recommended pattern:

<client-engagement>/reports/
├── 2026-Q1-baseline.pbix
├── 2026-04-25-after-workshop.pbix
└── 2026-05-10-pre-cutover.pbix

Each .pbix embeds the data at the time of save, so older snapshots remain reproducible even if the underlying export directories are rotated or pruned.

The .pbit template itself stays in swao\docs\templates\powerbi\ under git -- one canonical version per template-schema generation.
